When the user presents an idea, feature, or change request, follow this brainstorming workflow before any implementation:

1. **Explore project context** — check files, docs, recent commits
2. **Ask clarifying questions** — one at a time. Prefer multiple choice.
3. **Propose 2-3 approaches** — with trade-offs and your recommendation
4. **Present the design** — cover architecture, components, data flow, error handling, testing. Get user approval.
5. **Write the design doc** — save to `docs/opencode-skills/specs/YYYY-MM-DD-<topic>-design.md`
6. **Get user approval** — ask the user to review the spec before proceeding
7. **Transition** — invoke the writing-plans skill to create an implementation plan

**Critical rules:**
- NEVER write code, scaffold projects, or take implementation action before presenting and getting approval on a design
- This applies to EVERY project regardless of perceived simplicity
- One question at a time only
- The ONLY skill you invoke after brainstorming is writing-plans (not designing-frontend-interfaces, building-mcp-servers, etc.)
- Design for isolation: small units with one clear purpose, well-defined interfaces, independently testable

## Behavioral Guidelines

### Think Before Designing

**Don't assume. Don't hide confusion. Surface tradeoffs.**

- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations of the user's request exist, present them — don't pick silently.
- If a simpler approach than what the user envisions would solve the problem, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

### Simplicity First

**Minimum design that solves the problem. Nothing speculative.**

- No features in the design beyond what was asked.
- No "flexibility" or "configurability" that wasn't requested.
- If the design is complex, ask: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

### Goal-Driven Planning

**Define success criteria before implementation.**

When presenting the design, include verifiable success criteria:
- "Add validation" → success = "tests for invalid inputs pass"
- "Fix the bug" → success = "test that reproduces the bug passes"
- "Refactor X" → success = "tests pass before and after"

Strong success criteria let the build agent loop independently. Weak criteria ("make it work") require constant clarification.
