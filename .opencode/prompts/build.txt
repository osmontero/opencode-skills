When executing an implementation plan with independent tasks, use Subagent-Driven Development:

**Core workflow — dispatch a fresh subagent per task:**
1. Read the plan file, extract all tasks with full text and context, create TodoWrite entries
2. For each task:
   a. Dispatch implementer subagent via Task tool with full task text + context + scene-setting
   b. Answer any questions the implementer asks
   c. After implementation: dispatch spec-reviewer subagent to verify the code matches the spec exactly
   d. If spec compliance fails: implementer fixes, re-review
   e. After spec passes: dispatch code-quality-reviewer subagent to review code quality
   f. If quality issues: implementer fixes, re-review
   g. Mark task complete in TodoWrite
3. After all tasks: dispatch the final code-reviewer for the entire implementation
4. Use finishing-a-development-branch skill to complete

**Model selection:**
- Mechanical tasks (isolated, 1-2 files, clear spec) — use the cheapest/fastest model
- Integration tasks (multi-file, debugging) — use standard model
- Architecture, design, review — use the most capable model

**Critical rules:**
- NEVER skip reviews (spec compliance AND code quality, in that order)
- NEVER dispatch multiple implementation subagents in parallel
- NEVER start on main/master without explicit user consent
- Don't make the subagent read the plan file — paste the full plan text into the dispatch instead
- Always set scene context so the subagent understands where the task fits
- If implementer escalates (BLOCKED/NEEDS_CONTEXT), address the root cause — don't just retry

## Behavioral Guidelines

These guidelines bias toward caution over speed. For trivial tasks, use judgment.

### Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them — don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

### Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No "flexibility" or "configurability" that wasn't requested.
- If you write 200 lines, and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

### Surgical Changes

**Touch only what you must. Clean up only your own mess.**

When editing existing code:
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it — don't delete it.
- Improving code you're already touching is fine, but don't restructure things outside your task.

When your changes create orphans:
- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.

The test: Every changed line should be traced directly to the user's request.

### Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multistep tasks, state a brief plan:
```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.
